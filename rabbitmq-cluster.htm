What is cluster?

RabbitMQ cluster is a collection of nodes which shares the users, virtual hosts, exchanges and queues. By clustering, it can be ensured that RabbitMQ will be available without any downtime. In clustered setup, all the data is replicated across all the nodes. It has to be noted that the queues are not replicated across the nodes and if the node which is owning the queue goes down, there is no way the queue can be recovered, making the applications to throw "Queue not found" exception. To replicate the queues across the node, "HA" mode has to be enabled, which ensures that queues are replicated across the nodes and if the master goes down, the next node will be promoted as master and applications will start consuming from the queue in the new master node.

How to setup?

  • Install RabbitMQ in two or more servers. It is recommended to have the cluster formation across multiple instances because if one machine goes down, the other can take up the load.
  • Chose any one of the servers as the master node. Ideally, the server with bigger configuration

  • Copy the cookie from the master to all the slave nodes. The cookie can be found at

    cat /var/lib/rabbitmq/.erlang.cookie


  • Update `/etc/hosts` file with corresponding node names and add the details about the other nodes in the file. For example, the file in master node will look like:

    127.0.0.1   localhost4 localhost4localdomain4 master 
::1 localhost localhost.localdomain localhost6 localhost6.localdomain6 
172.31.24.147 slave


  • Update the node name in all the servers in the file /etc/rabbitmq/rabbitmq-env.conf

    • For example inmaster, the variable should be

      NODENAME=rabbit@master


  • Once this is done, kill the rabbitmq process in the nodes

  • Now, run the following commands in all the slave nodes. Note: $MASTER_HOSTNAME, should be changed to the value mentioned in the /etc/hosts for the master node

    sudorabbitmqctl stop_app;
sudo rabbitmqctl join_cluster rabbit@$MASTER_HOSTNAME;
sudo rabbitmqctl start_app;


  • Check the cluster status in the master node, 'rabbitmqctl cluster_status'. This should return all the nodes within the cluster. The cluster nodes will be listed management plugin as well.

  • Incase, if the user or virtual host is removed, run the following commands:

    rabbitmqctl add_user mobiquity mobiquity
rabbitmqctl set_user_tags mobiquity administrator
rabbitmqctl add_vhost /mobiquity_money
rabbitmqctl set_permissions -p /mobiquity_money mobiquity ".*" ".*" ".*"


High Availability mode (HA):

As mentioned earlier, the queues in the nodes are not replicated across the nodes. The queue is owned by the node where it is declared first by the application. When the node which owns the queue goes node, the application will throw up the exception as it is not able to connect to the queue. If the HA mode is enabled, the queues are replicated across all the nodes. When the master node goes down, the long running node will be promoted as master and the application will automatically switched to consume messages from the new node. There are chances that the messages can be re-delivered again to the application. So, the application should be made idempotent (no matter what, how many times it receives the message, it should produce the same result).

How to enable HA?

Enabling HA policy is very simple. It can be enabled using management plugin or through a command.   

Follow the steps below in the UI:

  • Select Admin tab
  • Choose Policies in the right tab
  • Under Add/Update Policy section
    • Select the virtual host

    • Give the name as "ha-all"

    • Leave the pattern as empty. This means that ha mode is enabled for all the queues
    • Add 'ha-mode=all' as definition.
  • Now check the queues, the policy will be added to all the queues

How to configure in application?

rabbitmq.addresses=172.31.31.156:5672,172.31.31.157:5672

The application establish the connection with the first host and the spring takes care switching the connection to the other node.
